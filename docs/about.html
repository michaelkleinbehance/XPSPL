About

prggmr is much more than just another framework, prggmr is a high-performance event environment for php, depending on the context of which you are developing software with prggmr will dictact how you interact with it, also for the record prggmr is to be spelled using all lowercase characters no exceptions.

I spent 4 years perfecting prggmr and in the end it turned out to be the exact opposite of what I envisioned and the exact of which I thought was impossible.

What

As previously mentioned prggmr is an event environment which has been developed for use with the PHP Programming language, it provides the ability for developing PHP applications using a complete Event Driven Architecture in the sense of both "Web" and "System" applications.

prggmr uses a hybrid of Signal Programming and the Observer Pattern for event processing allowing for developing PHP applications in ways that have only been dreamed of.

The most important aspect of prggmr is that it allows for developing PHP applications using a more natural approach, completely decoupled from themselves, simply meaning you can modify a applications flow, structure, data and architecture without modifying pre-existing code.

How

The most important interesting aspect, prggmr as previously mentioned uses a hybrid of Signals and the Observer Pattern, I consider it a hybrid since it doesn't fully adopt the methodologies of either but instead using only bits and pieces.

The basics of prggmr are detailed below.

Engine

prggmr uses a finely tuned processing engine to provide it's event architecture, the engine handles everything that happens in prggmr from subscribing to debugging those misbehaving subscriptions.

Queue

A priority based queue is used for handling subscriptions, it dictates the order in which subscriptions fire, is the "Observer" part of prggmr and represents a signal.

Signal

The "Signal" part of prggmr and the object which tells the engine what Queue to fire, the signal can be thought of as the object which introduces the change.

Subscription

The subscription is the function which is fired from within the Queue when the engine tells it to do so.

Event

The event is what the engine uses as the change representation, the event is usually automatically generated by the engine and passed around to subscriptions like a batton in a marathon race.

Now for the confusing stuff

Throughout the entire flow each Signal, Subscription and Event belong to nothing they are basically free, each knows nothing about the other and no direct line of communication is ever established between them, prggmr contains no circular object references and other than the Event having reference to the signal and subscription which triggered it, which is never guaranteed, each object is truly its own.

If you are no closer to understanding prggmr than when you began reading this page then don't worry, prggmr has been built so you need no knowledge of how an event architecture works as prggmr does all of the heavy lifting, you just need to learning a couple of simple functions and your ready to go.
